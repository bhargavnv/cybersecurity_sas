Substation Automation Systems and Cybersecurity through functional domains, profiles, and assurance

This module is inspired by the goals and requirements of IEC61850, IEC62443 and IEC62351 as the foundation and treats cybersecurity as an operational discipline attached to substation function, lifecycle state, and authority. It does not teach compliance itself, but rather an approach to the compliance.

MonitoringControlConfigurationEngineering

Primary objective: explain the substation through its functional domains, profile what is allowed in each, map the cyber problem to those domains, then show how IEC 62443, IEC 62351, and an assurance-oriented derivation model can be used to build controls that are both transparent and defensible.

Module 1 · Foundation

What we are is really trying to fix

Problem statement

The practical utility problem: heterogeneous, multi-vendor, brownfield-heavy systems still dominate, yet operational expectations now include remote visibility, coordinated control, auditability, distributed integration, and secure modernization. Protocol conformance alone does not solve inconsistent implementation, exposed endpoints, or decisions taken without lifecycle context.

Core approach

The core idea is to separate operational authority from communication transport. Native protocols and device behavior remain where they are, but external interactions are governed through a mediated interaction layer that is policy-driven, state-aware, and auditable.

Plain English: being able to reach a device should not automatically mean being allowed to affect it.

Module 1 · Motivation

Why digital substations create a richer cyber problem than wired substations

What changed technically

  • Logic moved from hardwired panels into IEDs, configurations, datasets, reports, and software-defined behaviour
  • Time synchronization, publisher-subscriber traffic, engineering files, and remote clients became operational dependencies
  • The station became an interaction system, not only a wiring system

What changed operationally

  • Remote monitoring and control became normal
  • Change velocity increased because settings, files, software, and integrations now matter
  • Evidence, traceability, and explainability became necessary for operations, compliance, and incident response

What changed in risk

  • A wrong value can mislead even without tripping anything
  • A benign maintenance path can become a live attack path
  • Configuration mistakes can become tomorrow’s runtime failure

Module 1 · Architecture view

A SAS is best understood through function, not by memorizing device names

Physical / communication view

  • Process level: instrument transformers, breakers, Merging Units, sensors, actuators
  • Bay level: protection and bay-control IEDs
  • Station level: HMI, gateway, historian, engineering access, local servers
  • External interface: control centre, enterprise, remote maintenance

Functional view used in this course

  • Monitoring: make system state visible
  • Control: change system state safely
  • Configuration: modify governed runtime parameters
  • Engineering: create or handle artifacts that shape future runtime behaviour

Why the functional view matters

The same physical component can participate in more than one domain. A relay is not just a relay. It can be a monitor, a controller, a configuration target, and an engineering endpoint depending on the interaction. Humans love pretending boundaries are obvious. The plant does not indulge them.

Module 1 · IEC 61850 context

Where IEC 61850 fits in this story

Functional relevance

  • IEC 61850 gives data models and service semantics that let utilities describe and exchange substation information consistently
  • MMS commonly supports client-server supervisory exchanges
  • GOOSE supports fast event-oriented publisher-subscriber messaging
  • Sampled Values carry time-sensitive process measurements

Cyber relevance

  • If these exchanges are operationally meaningful, then their misuse is also operationally meaningful
  • Security is not only about packet secrecy; it is about preserving semantics, timing, authority, and state correctness
  • The same protocol service can be acceptable in one domain and dangerous in another depending on purpose and lifecycle state

Module 1 · Monitoring

Monitoring domain in standard terms

How to understand the standard

  • be primarily stateless
  • use HTTP-based realization for on-demand data retrieval
  • support hierarchical and structured access to monitored values
  • support bulk and selective data access mechanisms
  • allow dataset-based and aggregated retrieval
  • enable filtering, pagination, and scope limitation
  • support caching and refresh strategies where appropriate
  • avoid excessive polling or performance impact on time-critical functions

What that means operationally

This domain is primarily about getting the right visibility without disturbing critical execution. It favours on-demand reads, structured access, filtering, and selective scope, and it must cooperate with event/report mechanisms for continuous observation.

Module 1 · Monitoring

Monitoring invariants and profile assumptions

Key invariants

  • The following invariants shall apply:: Semantic preservation Monitoring data shall remain aligned with the semantics and structure defined in IEC 61850 and related domain standards.
  • None: Non-intrusiveness Monitoring shall not degrade or affect operational or protection performance.
  • None: Deterministic access Data access behavior shall be predictable and bounded.
  • None: Consistency Identical monitoring profiles shall produce consistent results.
  • None: Selective access Access shall be limited to authorized scope and roles.
  • None: Context awareness Monitoring shall reflect system state and lifecycle context.

Profile assumptions to make explicit

  • Observation must not affect protection or control performance
  • Returned values retain IEC 61850 meaning, quality, and context
  • Scope of visibility is role-bound and asset-bound
  • Freshness, ordering, and source trust matter for operator decisions

Module 1 · Control

Control domain in standard terms

How to understand the standard

  • be stateful
  • use WebSocket-based realization for control workflows and feedback
  • preserve established control procedures including select-before-operate
  • support confirmation, cancellation, and timeout behavior
  • enable operator-driven and automated control
  • support policy-driven and context-aware decision-making
  • avoid interference with protection, safety, or bay-local automation
  • ensure bounded latency and deterministic response

What that means operationally

This domain is about governed state change, not about raw command transport. It expects multi-step procedures, confirmation and timeout behaviour, bounded latency, and the preservation of safety and interlocking rules.

Module 1 · Control

Control invariants and profile assumptions

Key invariants

  • The following invariants shall apply:: Semantic preservation Control semantics shall remain aligned with IEC 61850 and related domain workflows.
  • None: Workflow integrity Control shall preserve multi-step procedures such as select-before-operate.
  • None: Authorization and governance Control authority shall be identity-bound, role-based, and policy governed.
  • None: State awareness All control actions shall be evaluated against current system state, lifecycle mode, and safety constraints.
  • None: Safety preservation Control shall not bypass interlocking, protection, or safety mechanisms.
  • None: Determinism and bounded behavior Control execution and feedback shall be predictable and traceable.

Profile assumptions to make explicit

  • Workflow matters as much as command content
  • Authority must be identity-bound, role-bound, and context-bound
  • Current plant state and lifecycle mode constrain what is legal
  • Control must remain compatible with safety, interlocking, and local autonomy

Module 1 · Configuration

Configuration domain in standard terms

How to understand the standard

  • support structured and staged configuration workflows
  • use HTTP for discovery, validation, and staging
  • use WebSocket for controlled activation, coordination, and feedback where required
  • enable partial, incremental, and bulk updates
  • support versioned and traceable parameter changes
  • ensure safe interaction with operational and protection functions
  • avoid direct or uncontrolled modification of active system behavior
  • Configuration interactions shall not interfere with protection, local automation, or operational continuity

What that means operationally

This domain treats change as a governed workflow. Discovery, staging, validation, activation, rollback, and version control are part of the interaction, because “write succeeded” is not the same as “change was safe.”

Module 1 · Configuration

Configuration invariants and profile assumptions

Key invariants

  • The following invariants shall apply:: Semantic preservation Configuration semantics shall remain aligned with IEC 61850 and domain-specific engineering workflows.
  • None: Lifecycle awareness Configuration shall be evaluated against operational, maintenance, commissioning, and test states.
  • None: Staging and validation Configuration changes shall be staged and validated before activation.
  • None: Authorization and governance Changes shall be identity-bound, role-based, and policy governed.
  • None: Safety preservation Configuration shall not introduce unsafe or inconsistent operational states.
  • None: Rollback and reversibility All configuration changes shall support rollback and recovery.

Profile assumptions to make explicit

  • Changes are not safe merely because a tool can write them
  • Staging and validation happen before activation
  • Normal, maintenance, test, and commissioning states are different risk worlds
  • Every change requires recovery logic, version control, and attributable history

Module 1 · Engineering

Engineering domain in standard terms

How to understand the standard

  • Artifact, file, and software management interactions shall support structured and unstructured artifacts required for engineering, configuration, maintenance, and lifecycle management
  • support both structured and opaque artifacts
  • use HTTP for discovery, upload, download, and staging

Module 1 · Engineering

Engineering invariants and profile assumptions

Key invariants

  • The following invariants shall apply:: Inclusiveness Both structured and proprietary artifact formats shall be supported.
  • None: Opaque governance Proprietary or byte-level artifacts shall be governed even when semantics are unknown.
  • None: Integrity assurance All artifacts shall support validation using checksums, signatures, or equivalent mechanisms.
  • None: Traceability Artifact origin, version, and lifecycle shall be recorded.
  • None: Staging and validation Artifacts shall be staged and validated before activation.
  • None: Lifecycle awareness Artifact usage shall depend on system lifecycle state.

Profile assumptions to make explicit

  • Opaque artifacts still need provenance and integrity control
  • Toolchain trust cannot be assumed forever
  • Artifact handling depends on lifecycle state and approval context
  • Previous known-good artifacts must remain recoverable

Module 1 · Governing context

Lifecycle and mode management governs all four domains

Standard position

  • Lifecycle and mode management defines and governs the operational state of systems, devices, and functions
  • It acts as a governing context for all other functional areas
  • Mode discovery, transition requests, coordinated transition, and monitoring must be safe, authorized, and traceable

State is not decoration

Why it matters to cyber

  • The same interaction can be acceptable in maintenance and forbidden in normal service
  • Temporary permissions without explicit state handling turn into silent permanent risk
  • Most ugly incidents involve some confusion between intended state and actual state

Working set

Useful state labels for teaching

  • Normal service
  • Fault / disturbance response
  • Maintenance
  • Commissioning / testing

Module 1 · Supporting domains

Identity, trust, and audit are not side dishes

Identity and authorization context

  • Authentication is not authorization
  • Authorization must consider state, role, purpose, and interaction scope
  • Trust should be dynamic and revocable, not static and ceremonial

Audit, traceability, and evidence

  • Critical interactions must be attributable to identity, role, context, and time
  • Timestamps, ordering, and tamper resistance matter
  • Audit must work transparently and should not disturb protection or operations

Teaching implication

Whenever you explain any substation cyber incident, ask: who was the actor, what was the state, what was the scope, what evidence survived, and who could later reconstruct the chain? If those questions have mushy answers, you already have a design problem.

Module 1 · Cyber problem

Why cybersecurity emerges naturally from SAS, even before an attacker appears

Deliberate pathways

  • Unauthorized remote access or misuse of legitimate paths
  • Spoofed, replayed, or manipulated communications
  • Compromised engineering workstations, artifacts, or maintenance channels
  • Insider misuse or supplier access abuse

Inadvertent pathways

  • Wrong settings or wrong files
  • Improper state transitions or maintenance left active
  • Excessive polling, malformed subscriptions, network loops, and accidental overload
  • Poor rollback, stale baselines, or inconsistent versions across devices

Key point

Both deliberate and inadvertent pathways violate the same kinds of assumptions: who may interact, how, in what state, within what timing bounds, and with what evidence trail. That is why a good security model also improves reliability and maintainability.

Module 1 · Threat types

Threat categories to use during delivery

Use a simple first split

  • Deliberate threats: adversarial, deceptive, exploitative, privilege-seeking
  • Inadvertent threats: error, omission, misunderstanding, drift, overload, poor lifecycle discipline

Use frameworks, do not worship them

Then add a modelling stack

  • STRIDE for broad threat classes on systems and interactions
  • MITRE ATT&CK for ICS for attacker procedures and TTPs
  • LINDDUN where monitoring and data visibility introduce privacy/governance questions
  • U-OAMM for deriving controls from assumptions, behaviours, risks, zones, conduits, and evidence

Why this stack works

STRIDE tells you the kind of failure. MITRE tells you how it may be executed. U-OAMM tells you how to derive a substation-worthy response instead of collecting buzzwords like trading cards.

Module 1 · Threat modelling

Threat-modelling lenses and what each adds

STRIDE

  • Spoofing: false identity or false source
  • Tampering: unauthorized modification of messages, settings, or artifacts
  • Repudiation: inability to prove who did what
  • Information disclosure: visibility beyond intended scope
  • Denial of service: loss of timely function or overload
  • Elevation of privilege: action beyond allowed authority

MITRE ATT&CK for ICS

  • Gives procedure language: initial access, execution, persistence, evasion, impairment of process control, inhibition of response function and more
  • Useful for case studies because it describes how adversaries actually move through ICS environments

U-OAMM

  • Start from assumptions, not from control checklists
  • Derive required behaviour, then risk, then grouping, zones, conduits, controls, and assurance questions
  • Keeps the model grounded in utility process consequences

Module 2 · U-OAMM

Use the derivation pipeline instead of jumping straight to controls

Derivation pipeline

Assumptions → Behavior → Risk → Zones → Conduits → Controls → Assurance

The model says zoning, conduits, and controls are outcomes. If they are not derived from explicit assumptions and behaviour, they become decorative confidence.

Why this matters in substations

  • Protection, control, visibility, maintenance, and engineering are stateful and time-bounded
  • Missing required behaviour is as dangerous as extra forbidden behaviour
  • Controls should preserve engineering constraints rather than quietly becoming new plant dependencies

Isolate control paths into tightly bounded conduits; Require explicit role separation between observation, authorization, and execution where consequence is high; Use defence in depth so a remote path compromise cannot directly negate bay safety logic
IEC 62351 contribution
Protect session identity and protocol exchanges used for supervisory control and MMS-based operations; Use cryptographic trust and secure channel handling where the protocol profile supports it; Preserve integrity and authenticity of control-related communications and status feedback
Beyond the standards
Two-person control for high-consequence actions; State-bound permissions and just-in-time approvals; Command simulation / dry-run explanation before operate; Operator intent capture and post-action reconciliation
Transparency requirement
Control should make it obvious who authorized, initiated, confirmed, and observed the action, under which state and safety context.

Module 2 · Shared controls

Identity, lifecycle, and audit are the glue across all domains

Identity and authorization

  • Identity alone is insufficient
  • Authorization must be context-bound
  • Roles, intent, asset scope, lifecycle state, and time window all matter

Lifecycle and state

  • Normal, maintenance, commissioning, and recovery states require different permissions and controls

  • Temporary access without expiry becomes permanent attack surface

  • Audit should support incident reconstruction and operational explanation
  • Transparent controls are easier to audit because their logic is explicit

Module 2 · IEC 62443

What IEC 62443 contributes to a substation cybersecurity design

What it gives you

  • A lifecycle-oriented security framework for industrial automation and control systems
  • Zones and conduits as containment structure
  • Security requirements organized by system, component, and process perspectives
  • A way to reason about security levels and defence in depth

How to use it here

  • Do not start by drawing zones from the network diagram alone
  • Use the functional-domain risks to justify zones and conduits
  • Apply lifecycle discipline to suppliers, maintenance, account use, and change processes
  • Keep conduits minimal, directional, and state-bounded

What it does not magically do

IEC 62443 does not automatically understand substation semantics, select-before-operate workflow, or the meaning of protection and engineering artifacts. You still have to derive those from the plant. Standards are frameworks, not user guides.

Module 2 · IEC 62351

What IEC 62351 contributes to a power-system-specific design

What it covers

  • Security technologies for IEC TC57 communication protocols
  • Measures for authenticated access, confidentiality where needed, integrity protection, and secure protocol handling
  • Guidance across protocols used in grid and utility control communications

How to use it here

  • Apply it where standardized power-system communications are actually in play
  • Tie protocol security to the functional domain and risk, not just to the existence of a protocol
  • Remember that secure channel protection does not replace workflow, lifecycle, or provenance governance

Useful teaching simplification

  • 62443 answers: how should the industrial system be structured and managed?
  • 62351 answers: how should key power-system communications be protected?
  • U-OAMM answers: why this control, for this behaviour, in this state, with this evidence?

Module 2 · Beyond standards

What to do beyond 62443 and 62351 for genuine peace of mind

Operational peace of mind means

  • You can explain why an action was allowed or denied
  • You can reconstruct what happened without improvisation
  • You can detect drift in assumptions before process harm
  • You can recover to a known-good state without guesswork

Practical additions

  • State-aware just-in-time access
  • Two-person control for high consequence actions
  • Golden baselines and compare-before-activate
  • Artifact provenance ledger and independent verification
  • Independent corroboration for critical visibility paths
  • Assumption validation schedules, not one-time paperwork

Non-negotiable attitude

Peace of mind is not a feeling. It is the ability to answer assurance questions quickly with evidence. Anything else is just expensive optimism in a rack.

Module 2 · Transparency

What transparent controls look like in practice

A transparent control should reveal

  • Which interaction it governs
  • Which role and identity it depends on
  • Which state makes it legal or illegal
  • Which risk statement it addresses
  • Which evidence proves it worked

A non-transparent control usually looks like

  • A broad allow rule justified by convenience
  • A supplier exception nobody can defend later
  • A workflow hidden inside one workstation or one engineer’s memory
  • Audit that records events but not decisions or context

Why transparency matters

Transparent controls are easier to operate, easier to audit, easier to review with vendors, and easier to improve after incidents. They also make the training credible because participants can see the control logic instead of memorizing slogans.

Close the loop with assurance questions

Assumptions

  • Which assumptions does this station rely on today?
  • Which assumptions are validated continuously and which are not?
  • How is assumption drift detected before process impact?

Behaviours and risks

  • Are only required or acceptable behaviours present?
  • Are timing and sequence bounds preserved?

Controls and evidence

  • Is every important control mapped to a risk statement?
  • Can you explain every cross-zone behaviour without hand-waving?

Final take-away

A secure digital substation is not one with the most cyber products attached to it. It is one where each functional domain has a clear profile, each interaction is governed in context, each cross-boundary behaviour is justified, and each important decision leaves evidence behind.

That is how IEC 61850 functional thinking, IEC 62443 architectural discipline, IEC 62351 protocol protection, and an assurance-oriented derivation model can be taught as one coherent story rather than as four different standards lectures pretending to be friends.
